SP7 整体架构 — 3 种方案对比

Skill 系统 + 自进化(4 维度)。请选择整体架构走向

A

方案 A:最小集成

Skill 注入: 启动时拼到 system_prompt 末尾,用 <skill> 标签包裹

进化维度: 仅 prompt 优化(1/4)

存储: AgentLibrary 直接覆盖,无版本

触发: 用户手动 POST /evolve

优势: 改动最小、1 个 SP 完成

劣势: 不满足用户"4 维度 + 自动触发"需求

B

方案 B:解耦版(推荐)

Skill 注入: TeamCompiler 编译期加载 skill .md,作为独立 SystemMessage 注入 react_messages 首部(与 system_prompt 解耦)

进化维度: 全 4 维度(prompt / 参数 / skill 生成 / 动态 skill 选择)

存储: Agent 加 version 字段 + 新增 evolution_history 表(before/after/diff/reason)

触发: Run 结束后异步自动触发(不阻塞 API 响应)

SP 拆分: SP7a(skill 系统)+ SP7b(自进化),分阶段实施

优势: 满足全部需求、skill 与 prompt 解耦便于独立演进、版本化可回滚

劣势: 改动较大、需要 2 个 SP 周期

C

方案 C:RAG 增强版

Skill 注入: Skill 库 + 向量索引,Agent 每次迭代按任务描述检索 top-K skill,动态注入

进化维度: 全 4 维度 + skill 自动标注/聚类

存储: SQLite skill 元数据 + 向量数据库(FAISS/Chroma)

触发: Run 结束后自动 + 周期性 skill 重排序

优势: Skill 选择智能化、可扩展到大 skill 库(1000+)

劣势: 引入向量数据库依赖、复杂度高、调试困难、过度工程化

💡 推荐:方案 B

理由:

  1. 满足全部 4 维度需求,同时通过 SP 拆分控制单次实施范围
  2. Skill 作为 SystemMessage 注入比拼到 system_prompt 更清晰:LLM 把 skill 视为"行为指导"而非"身份描述",且 prompt 长度可控
  3. 版本化 history 表是安全网:进化退化时可回滚,生产环境必需
  4. 异步自动触发满足用户需求,但通过"不阻塞 API + 失败保护"控制风险
  5. 方案 C 的 RAG 在 skill 数量 < 100 时收益有限,反而增加运维负担。若后期 skill 库做大可平滑升级
