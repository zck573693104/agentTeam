# Error Handling Skill

当任务涉及错误处理设计或代码编写时,遵循以下原则:

## 1. 异常分类
- **可恢复异常**(网络超时、资源占用):用重试或降级处理
- **不可恢复异常**(配置错误、数据损坏):快速失败,记录日志
- **业务异常**(用户输入错误、权限不足):转成用户友好消息

## 2. 重试策略
- 指数退避:1s → 2s → 4s → 8s,最多 4 次
- 区分可重试错误(5xx、超时)与不可重试错误(4xx、参数错误)
- 重试失败后,落库 + 告警,不静默吞没

## 3. 错误传播
- 跨层调用:用领域异常封装底层异常,保留 cause
- API 边界:把内部异常转成 HTTP 状态码 + 标准错误响应
- 不要 catch Exception 后什么都不做(吞没 = 调试噩梦)

## 4. 日志规范
- error 级别:系统故障,需人工介入
- warn 级别:可恢复异常,自动处理后继续
- info 级别:业务异常,记录用户操作上下文
- 日志含:trace_id / user_id / 关键参数 / 异常堆栈

## 5. 用户消息
- 不暴露技术细节(如 SQL 错误、堆栈信息)
- 给出可操作的建议("请检查网络后重试"而非"connection refused")
- 业务异常用用户能理解的语言(如"余额不足"而非"INSUFFICIENT_BALANCE")
